# <a name="monolithic-applications"></a><span data-ttu-id="257fc-103">整合型應用程式</span><span class="sxs-lookup"><span data-stu-id="257fc-103">Monolithic applications</span></span>

<span data-ttu-id="257fc-104">在本案例中，您會建置單一且整合型的 Web 應用程式或服務，並將它部署為容器。</span><span class="sxs-lookup"><span data-stu-id="257fc-104">In this scenario, you're building a single and monolithic web application or service and deploying it as a container.</span></span> <span data-ttu-id="257fc-105">從應用程式內部來看，其結構可能不是整合型，而是由數個程式庫、元件，甚至層級 (應用程式層、網域層、資料存取層等) 所組成。</span><span class="sxs-lookup"><span data-stu-id="257fc-105">Within the application, the structure might not be monolithic; it might comprise several libraries, components, or even layers (application layer, domain layer, data access layer, etc.).</span></span> <span data-ttu-id="257fc-106">從外部來看，它是單一容器，像是單一處理序、單一 Web 應用程式或單一服務。</span><span class="sxs-lookup"><span data-stu-id="257fc-106">Externally, it's a single container, like a single process, single web application, or single service.</span></span>

<span data-ttu-id="257fc-110">當您遵循「容器僅執行一項動作並在一個處理序中執行該動作」的原則時，整合型模式可能會產生衝突。</span><span class="sxs-lookup"><span data-stu-id="257fc-110">Following the principal that a container does one thing only, and does it in one process, the monolithic pattern is in conflict.</span></span> <span data-ttu-id="257fc-111">您可以在每個容器中包含多個元件/程式庫或內部層級，如圖 4-1 所示。</span><span class="sxs-lookup"><span data-stu-id="257fc-111">You can include multiple components/libraries or internal layers within each container, as illustrated in Figure 4-1.</span></span>

<span data-ttu-id="257fc-113">**圖4-1。**</span><span class="sxs-lookup"><span data-stu-id="257fc-113">**Figure 4-1.**</span></span> <span data-ttu-id="257fc-114">整合型應用程式的架構範例</span><span class="sxs-lookup"><span data-stu-id="257fc-114">An example of monolithic application architecture</span></span>

<span data-ttu-id="257fc-115">整合型應用程式的單一處理序或容器內即具有其全部或大部分功能，且在內部層或程式庫中經元件化處理。</span><span class="sxs-lookup"><span data-stu-id="257fc-115">A monolithic app has all or most of its functionality within a single process or container and it's componentized in internal layers or libraries.</span></span> <span data-ttu-id="257fc-116">如果應用程式因成長而需要擴充，則此方法的缺點便會浮現。</span><span class="sxs-lookup"><span data-stu-id="257fc-116">The downside to this approach comes if or when the application grows, requiring it to scale.</span></span> <span data-ttu-id="257fc-117">若整個應用程式都擴充，則不成問題。</span><span class="sxs-lookup"><span data-stu-id="257fc-117">If the entire application scaled, it's not really a problem.</span></span> <span data-ttu-id="257fc-118">不過，在大多數情況下，應用程式只需要擴充幾個造成阻礙的部分，其他元件則較少使用。</span><span class="sxs-lookup"><span data-stu-id="257fc-118">However, in most cases, a few parts of the application are the choke points that require scaling, whereas other components are used less.</span></span>

<span data-ttu-id="257fc-119">若是一般的電子商務範例，您可能需要調整的是產品資訊元件。</span><span class="sxs-lookup"><span data-stu-id="257fc-119">Using the typical e-commerce example, what you likely need is to scale the product information component.</span></span> <span data-ttu-id="257fc-120">瀏覽產品的客戶比購買的人多。</span><span class="sxs-lookup"><span data-stu-id="257fc-120">Many more customers browse products than purchase them.</span></span> <span data-ttu-id="257fc-121">比起使用付款管道，會有更多客戶使用其購物籃。</span><span class="sxs-lookup"><span data-stu-id="257fc-121">More customers use their basket than use the payment pipeline.</span></span> <span data-ttu-id="257fc-122">新增留言或檢視其購買歷程記錄的客戶較少。</span><span class="sxs-lookup"><span data-stu-id="257fc-122">Fewer customers add comments or view their purchase history.</span></span> <span data-ttu-id="257fc-123">而且在單一區域中，您可能只有少數幾個員工來管理內容和行銷活動。</span><span class="sxs-lookup"><span data-stu-id="257fc-123">And you likely have only a handful of employees, in a single region, that need to manage the content and marketing campaigns.</span></span> <span data-ttu-id="257fc-124">藉由擴充整合型設計，即可多次部署所有程式碼。</span><span class="sxs-lookup"><span data-stu-id="257fc-124">By scaling the monolithic design, all of the code is deployed multiple times.</span></span>

<span data-ttu-id="257fc-125">除了「全部調整」的問題之外，當您變更單一元件時，必須徹底重新測試整個應用程式，並重新部署所有執行個體。</span><span class="sxs-lookup"><span data-stu-id="257fc-125">In addition to the "scale-everything" problem, changes to a single component require complete retesting of the entire application as well as a complete redeployment of all the instances.</span></span>

<span data-ttu-id="257fc-126">整合型方法很常見，許多組織也開始使用這個架構方法進行開發。</span><span class="sxs-lookup"><span data-stu-id="257fc-126">The monolithic approach is common, and many organizations are developing with this architectural method.</span></span> <span data-ttu-id="257fc-127">其中有許多組織獲得良好的結果，但其他組織卻遭遇到限制。</span><span class="sxs-lookup"><span data-stu-id="257fc-127">Many enjoy good enough results, whereas others encounter limits.</span></span> <span data-ttu-id="257fc-128">由於工具和基礎結構太難建置 SOA，因此許多組織按照此模型來設計應用程式，但是在應用程式成長之後需求才會顯現出來。</span><span class="sxs-lookup"><span data-stu-id="257fc-128">Many designed their applications in this model because the tools and infrastructure were too difficult to build SOAs, and they didn't see the need—until the app grew.</span></span>

<span data-ttu-id="257fc-129">從基礎結構的觀點來看，每部伺服器都可以在相同主機內執行許多應用程式，並具備可接受的資源使用效率比，如圖 4-2 所示。</span><span class="sxs-lookup"><span data-stu-id="257fc-129">From an infrastructure perspective, each server can run many applications within the same host and have an acceptable ratio of efficiency in your resources usage, as shown in Figure 4-2.</span></span>

<span data-ttu-id="257fc-133">最後，從可用性的觀點來看，整合型應用程式必須以整體方式部署；這表示萬一您必須「停止和啟動」時，會影響部署期間的所有功能和所有使用者。</span><span class="sxs-lookup"><span data-stu-id="257fc-133">Finally, from an availability perspective, monolithic applications must be deployed as a whole; that means that in case you must *stop and start*, all functionality and all users will be affected during the deployment window.</span></span> <span data-ttu-id="257fc-134">在某些情況下，使用 Azure 和容器可以盡量避免這些情況，並降低應用程式停機的可能性，如圖 4-3 所示。</span><span class="sxs-lookup"><span data-stu-id="257fc-134">In certain situations, the use of Azure and containers can minimize these situations and reduce the probability of downtime of your application, as you can see in Figure 4-3.</span></span>

<span data-ttu-id="257fc-137">您也可以使用 [Azure App Service](https://azure.microsoft.com/services/app-service/) 來執行整合型應用程式並輕鬆調整執行個體，而不必管理 VM。</span><span class="sxs-lookup"><span data-stu-id="257fc-137">You can also use [Azure App Services](https://azure.microsoft.com/services/app-service/) to run monolithic applications and easily scale instances without having to manage the VMs.</span></span> <span data-ttu-id="257fc-138">Azure App Service 也可以執行 Docker 容器的單一執行個體，以簡化部署。</span><span class="sxs-lookup"><span data-stu-id="257fc-138">Azure App Services can run single instances of Docker containers, as well, simplifying the deployment.</span></span>

## <a name="monolithic-application-deployed-as-a-container"></a><span data-ttu-id="257fc-146">整合型應用程式部署為容器</span><span class="sxs-lookup"><span data-stu-id="257fc-146">Monolithic application deployed as a container</span></span>

<span data-ttu-id="257fc-147">使用容器來管理整合型部署有許多優點。</span><span class="sxs-lookup"><span data-stu-id="257fc-147">There are benefits to using containers to manage monolithic deployments.</span></span> <span data-ttu-id="257fc-148">調整容器執行個體遠比部署額外的 VM 更輕鬆快速。</span><span class="sxs-lookup"><span data-stu-id="257fc-148">Scaling the instances of containers is far faster and easier than deploying additional VMs.</span></span>

<span data-ttu-id="257fc-149">以 Docker 映像來部署更新會更快且網路效率更高。</span><span class="sxs-lookup"><span data-stu-id="257fc-149">Deploying updates as Docker images is far faster and network efficient.</span></span> <span data-ttu-id="257fc-150">Docker 容器通常只要幾秒鐘就能啟動，因此可加速推出。</span><span class="sxs-lookup"><span data-stu-id="257fc-150">Docker containers typically start in seconds, speeding rollouts.</span></span> <span data-ttu-id="257fc-151">您只要叫用 `docker stop` 命令即可輕鬆卸除 Docker 容器，且通常不到一秒就會完成。</span><span class="sxs-lookup"><span data-stu-id="257fc-151">Tearing down a Docker container is as easy as invoking the `docker stop` command, typically completing in less than a second.</span></span>

<span data-ttu-id="257fc-152">容器的設計原本即為固定，因此您永遠不需要擔心會因為更新指令碼忘記處理部分特定設定或是檔案殘留在磁碟上而導致 VM 損毀。</span><span class="sxs-lookup"><span data-stu-id="257fc-152">Because containers are inherently immutable, by design, you never need to worry about corrupted VMs because an update script forgot to account for some specific configuration or file left on disk.</span></span>

<span data-ttu-id="257fc-153">雖然整合型應用程式可以從 Docker 獲益，但這只是一小部分優點。</span><span class="sxs-lookup"><span data-stu-id="257fc-153">Although monolithic apps can benefit from Docker, we're touching on only the tips of the benefits.</span></span> <span data-ttu-id="257fc-154">管理容器的主要優點來自於使用容器協調器進行部署，透過此方式來管理各種執行個體和每個容器執行個體的生命週期。</span><span class="sxs-lookup"><span data-stu-id="257fc-154">The larger benefits of managing containers come from deploying with container orchestrators that manage the various instances and life cycle of each container instance.</span></span> <span data-ttu-id="257fc-155">將整合型應用程式分成可個別擴充、開發及部署的多個子系統，是您開始使用微服務的不錯起點。</span><span class="sxs-lookup"><span data-stu-id="257fc-155">Breaking up the monolithic application into subsystems that can be scaled, developed, and deployed individually is your entry point into the realm of microservices.</span></span>

<span data-ttu-id="257fc-156">若要瞭解如何使用容器來「隨即轉移」整合型應用程式，以及如何將應用程式現代化，您可以閱讀這份額外的 Microsoft 指南、將 [現有的 .net 應用程式現代化到 Azure 雲端和 Windows 容器](../../modernize-with-azure-containers/index.md)，您也可以從下載 PDF <https://aka.ms/LiftAndShiftWithContainersEbook> 。</span><span class="sxs-lookup"><span data-stu-id="257fc-156">To learn about how to "lift and shift" monolithic applications with containers and how you can modernize your applications, you can read this additional Microsoft guide, [Modernize existing .NET applications with Azure cloud and Windows Containers](../../modernize-with-azure-containers/index.md), which you can also download as PDF from <https://aka.ms/LiftAndShiftWithContainersEbook>.</span></span>
