## <a name="principle-of-least-privilege"></a><span data-ttu-id="8aaa1-130">最低權限原則</span><span class="sxs-lookup"><span data-stu-id="8aaa1-130">Principle of least privilege</span></span>

<span data-ttu-id="8aaa1-131">電腦安全性性的其中一項成立概念是 (POLP) 的最低許可權原則。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-131">One of the founding ideas in computer security is the Principle of Least Privilege (POLP).</span></span> <span data-ttu-id="8aaa1-132">它實際上是以數位或實體形式的任何安全性形式的基本概念。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-132">It's actually a foundational idea in most any form of security be it digital or physical.</span></span> <span data-ttu-id="8aaa1-133">簡而言之，原則是任何使用者或進程都應該擁有最少的許可權來執行其工作。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-133">In short, the principle is that any user or process should have the smallest number of rights possible to execute its task.</span></span>

<span data-ttu-id="8aaa1-134">例如，您可以考慮在銀行 tellers：存取 safe 是不尋常的活動。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-134">As an example, think of the tellers at a bank: accessing the safe is an uncommon activity.</span></span> <span data-ttu-id="8aaa1-135">因此，平均的出納無法開啟安全的本身。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-135">So, the average teller can't open the safe themselves.</span></span> <span data-ttu-id="8aaa1-136">若要取得存取權，他們需要透過銀行經理來提升其要求，以執行額外的安全性檢查。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-136">To gain access, they need to escalate their request through a bank manager, who performs additional security checks.</span></span>

<span data-ttu-id="8aaa1-137">在電腦系統中，有一個很棒的例子，就是使用者連接到資料庫的許可權。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-137">In a computer system, a fantastic example is the rights of a user connecting to a database.</span></span> <span data-ttu-id="8aaa1-138">在許多情況下，都有一個用來建立資料庫結構和執行應用程式的單一使用者帳戶。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-138">In many cases, there's a single user account used to both build the database structure and run the application.</span></span> <span data-ttu-id="8aaa1-139">除非在極端的情況下，執行應用程式的帳戶不需要更新架構資訊的能力。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-139">Except in extreme cases, the account running the application doesn't need the ability to update schema information.</span></span> <span data-ttu-id="8aaa1-140">應該會有數個帳戶提供不同層級的許可權。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-140">There should be several accounts that provide different levels of privilege.</span></span> <span data-ttu-id="8aaa1-141">應用程式只能使用授與讀取和寫入存取權給資料表中之資料的許可權層級。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-141">The application should only use the permission level that grants read and writes access to the data in the tables.</span></span> <span data-ttu-id="8aaa1-142">這種保護措施可以消除目標是卸載資料庫資料表或引入惡意觸發程式的攻擊。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-142">This kind of protection would eliminate attacks that aimed to drop database tables or introduce malicious triggers.</span></span>

<span data-ttu-id="8aaa1-143">幾乎每個建立雲端原生應用程式的部分，都可以從記住最低許可權的原則來獲益。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-143">Almost every part of building a cloud-native application can benefit from remembering the principle of least privilege.</span></span> <span data-ttu-id="8aaa1-144">在以角色為基礎的存取控制中設定防火牆、網路安全性群組、角色和範圍 (RBAC) 時，可以找到它。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-144">You can find it at play when setting up firewalls, network security groups, roles, and scopes in Role-based access control (RBAC).</span></span>

## <a name="penetration-testing"></a><span data-ttu-id="8aaa1-145">滲透測試</span><span class="sxs-lookup"><span data-stu-id="8aaa1-145">Penetration testing</span></span>

<span data-ttu-id="8aaa1-155">Azure 本身會不斷地從 [Microsoft 內的駭客團隊](https://azure.microsoft.com/resources/videos/red-vs-blue-internal-security-penetration-testing-of-microsoft-azure/)進行攻擊。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-155">Azure itself is constantly undergoing attacks from a [team of hackers inside Microsoft](https://azure.microsoft.com/resources/videos/red-vs-blue-internal-security-penetration-testing-of-microsoft-azure/).</span></span> <span data-ttu-id="8aaa1-156">多年來，他們已經成為找出數十個潛在的災難性攻擊媒介的第一種，在外部遭到惡意探索之前先加以關閉。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-156">Over the years, they've been the first to find dozens of potentially catastrophic attack vectors, closing them before they can be exploited externally.</span></span> <span data-ttu-id="8aaa1-157">目標愈吸引人，永久性的動作專案會嘗試利用它，而世界上有幾個目標比 Azure 更有吸引力。</span><span class="sxs-lookup"><span data-stu-id="8aaa1-157">The more tempting a target, the more likely that eternal actors will attempt to exploit it and there are a few targets in the world more tempting than Azure.</span></span>
