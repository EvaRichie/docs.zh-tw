<span data-ttu-id="1a976-166">適用於實作抽象事件匯流排的傳訊技術是在不同層級。</span><span class="sxs-lookup"><span data-stu-id="1a976-166">The messaging technologies available for implementing your abstract event bus are at different levels.</span></span> <span data-ttu-id="1a976-167">例如，像是 RabbitMQ (一種傳訊訊息代理程式傳輸) 和 Azure 服務匯流排的產品，所在的層級便比其他產品低，例如 NServiceBus、MassTransit 或 Brighter，它們可以在 RabbitMQ 和 Azure 服務匯流排之上運作。</span><span class="sxs-lookup"><span data-stu-id="1a976-167">For instance, products like RabbitMQ (a messaging broker transport) and Azure Service Bus sit at a lower level than other products like, NServiceBus, MassTransit, or Brighter, which can work on top of RabbitMQ and Azure Service Bus.</span></span> <span data-ttu-id="1a976-168">您的選擇取決於您的應用程式需要多少功能和現成的延展性。</span><span class="sxs-lookup"><span data-stu-id="1a976-168">Your choice depends on how many rich features at the application level and out-of-the-box scalability you need for your application.</span></span> <span data-ttu-id="1a976-169">針對只為您的開發環境實作概念證明事件匯流排，如同我們在 eShopOnContainers 範例所做的事，在 Docker 容器上執行的 RabbitMQ 上進行簡單實作可能就足夠了。</span><span class="sxs-lookup"><span data-stu-id="1a976-169">For implementing just a proof-of-concept event bus for your development environment, as it was done in the eShopOnContainers sample, a simple implementation on top of RabbitMQ running on a Docker container might be enough.</span></span>

<span data-ttu-id="1a976-170">但對於需要高延展性的關鍵任務和生產系統，您可能想要評估 Azure 服務匯流排。</span><span class="sxs-lookup"><span data-stu-id="1a976-170">However, for mission-critical and production systems that need hyper-scalability, you might want to evaluate Azure Service Bus.</span></span> <span data-ttu-id="1a976-171">對於可讓您更輕鬆地開發分散式應用程式的高階抽象和功能，我們建議您評估其他商業和開放原始碼服務匯流排，例如 NServiceBus、 MassTransit 和 Brighter。</span><span class="sxs-lookup"><span data-stu-id="1a976-171">For high-level abstractions and features that make the development of distributed applications easier, we recommend that you evaluate other commercial and open-source service buses, such as NServiceBus, MassTransit, and Brighter.</span></span> <span data-ttu-id="1a976-172">當然，您可以在像是 RabbitMQ 和 Docker 的較低層級技術之上自行建立服務匯流排功能。</span><span class="sxs-lookup"><span data-stu-id="1a976-172">Of course, you can build your own service-bus features on top of lower-level technologies like RabbitMQ and Docker.</span></span> <span data-ttu-id="1a976-173">但是，那樣的配管工作對於自訂企業應用程式而言成本可能太高。</span><span class="sxs-lookup"><span data-stu-id="1a976-173">But that plumbing work might cost too much for a custom enterprise application.</span></span>

## <a name="resiliently-publishing-to-the-event-bus"></a><span data-ttu-id="1a976-174">彈性地發行到事件匯流排</span><span class="sxs-lookup"><span data-stu-id="1a976-174">Resiliently publishing to the event bus</span></span>

<span data-ttu-id="1a976-175">跨多個微服務實作事件驅動架構時，有一項挑戰是如何以不可分割方式更新原始微服務的狀態，同時以某種方式，根據交易彈性地將其相關整合事件發行到事件匯流排。</span><span class="sxs-lookup"><span data-stu-id="1a976-175">A challenge when implementing an event-driven architecture across multiple microservices is how to atomically update state in the original microservice while resiliently publishing its related integration event into the event bus, somehow based on transactions.</span></span> <span data-ttu-id="1a976-176">以下是一些完成這項功能的方法，雖然也有其他方法。</span><span class="sxs-lookup"><span data-stu-id="1a976-176">The following are a few ways to accomplish this functionality, although there could be additional approaches as well.</span></span>

- <span data-ttu-id="1a976-179">使用交易記錄採礦。</span><span class="sxs-lookup"><span data-stu-id="1a976-179">Using transaction log mining.</span></span>

- <span data-ttu-id="1a976-180">使用完整的[事件溯源模式](/azure/architecture/patterns/event-sourcing)。</span><span class="sxs-lookup"><span data-stu-id="1a976-180">Using full [Event Sourcing](/azure/architecture/patterns/event-sourcing) pattern.</span></span>

- <span data-ttu-id="1a976-181">使用[寄件匣模式](https://www.kamilgrzybek.com/design/the-outbox-pattern/)：交易式資料庫資料表作為訊息佇列，而訊息佇列將是事件建立者元件的基礎，會建立事件加以發行。</span><span class="sxs-lookup"><span data-stu-id="1a976-181">Using the [Outbox pattern](https://www.kamilgrzybek.com/design/the-outbox-pattern/): a transactional database table as a message queue that will be the base for an event-creator component that would create the event and publish it.</span></span>
