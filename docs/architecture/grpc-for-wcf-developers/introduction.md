## <a name="history"></a><span data-ttu-id="43bf4-114">歷程記錄</span><span class="sxs-lookup"><span data-stu-id="43bf4-114">History</span></span>

<span data-ttu-id="43bf4-115">電腦網路的基本原則，就像是一組電腦彼此交換資料，以達成一組相互關聯的工作在開始之後尚未改變。</span><span class="sxs-lookup"><span data-stu-id="43bf4-115">The fundamental principle of a computer network as nothing more than a group of computers exchanging data with each other to achieve a set of interrelated tasks hasn't changed since its inception.</span></span> <span data-ttu-id="43bf4-116">但複雜性、規模和期望以指數方式成長。</span><span class="sxs-lookup"><span data-stu-id="43bf4-116">But the complexity, scale, and expectations have grown exponentially.</span></span>

<span data-ttu-id="43bf4-117">在1990年代，重點在於改善使用相同語言和平臺的內部網路。</span><span class="sxs-lookup"><span data-stu-id="43bf4-117">During the 1990s, the emphasis was mainly on improving internal networks that used the same language and platforms.</span></span> <span data-ttu-id="43bf4-118">TCP/IP 成為這類通訊的金級標準。</span><span class="sxs-lookup"><span data-stu-id="43bf4-118">TCP/IP became the gold standard for this type of communication.</span></span>

<span data-ttu-id="43bf4-119">重點是藉由提升與語言無關的方法，讓您更有效地將跨多個平臺的通訊優化。</span><span class="sxs-lookup"><span data-stu-id="43bf4-119">The focus soon shifted to how best to optimize communication across multiple platforms by promoting a language-agnostic approach.</span></span> <span data-ttu-id="43bf4-120">服務導向的架構 (SOA) 提供了一種鬆散結合的結構，可提供給應用程式的廣泛服務集合。</span><span class="sxs-lookup"><span data-stu-id="43bf4-120">Service-oriented architecture (SOA) provided a structure for loosely coupling a broad collection of services that could be provided to an application.</span></span>

<span data-ttu-id="43bf4-121">當所有主要的平臺都可以存取網際網路時，就會發生 *web 服務* 的開發，但是它們仍無法彼此互動。</span><span class="sxs-lookup"><span data-stu-id="43bf4-121">The development of *web services* occurred when all major platforms could access the internet, but they still couldn't interact with each other.</span></span> <span data-ttu-id="43bf4-122">Web 服務有開放的標準和通訊協定，包括：</span><span class="sxs-lookup"><span data-stu-id="43bf4-122">Web services have open standards and protocols, including:</span></span>

- <span data-ttu-id="43bf4-123">XML 來標記和程式碼資料。</span><span class="sxs-lookup"><span data-stu-id="43bf4-123">XML to tag and code data.</span></span>
- <span data-ttu-id="43bf4-124">簡易物件存取通訊協定 (SOAP) 來傳輸資料。</span><span class="sxs-lookup"><span data-stu-id="43bf4-124">Simple Object Access Protocol (SOAP) to transfer data.</span></span>
- <span data-ttu-id="43bf4-125">Web 服務定義語言 (WSDL) 用來描述 web 服務並將其連接至用戶端應用程式。</span><span class="sxs-lookup"><span data-stu-id="43bf4-125">Web Services Definition Language (WSDL) to describe and connect web services to client applications.</span></span>
- <span data-ttu-id="43bf4-126"> (UDDI) 的通用描述、探索與整合，讓其他服務可以探索 web 服務。</span><span class="sxs-lookup"><span data-stu-id="43bf4-126">Universal Description, Discovery, and Integration (UDDI) to make web services discoverable by other services.</span></span>

<span data-ttu-id="43bf4-127">SOAP 會定義應用程式的分散式專案可以彼此通訊的規則，即使它們是在不同的平臺上也一樣。</span><span class="sxs-lookup"><span data-stu-id="43bf4-127">SOAP defines the rules by which distributed elements of an application can communicate with each other, even if they're on different platforms.</span></span> <span data-ttu-id="43bf4-128">SOAP 是以 XML 為基礎，因此它是人們可讀取的。</span><span class="sxs-lookup"><span data-stu-id="43bf4-128">SOAP is based on XML, so it's human-readable.</span></span> <span data-ttu-id="43bf4-129">容易理解 SOAP 的犧牲是大小;SOAP 訊息大於類似通訊協定中的訊息。</span><span class="sxs-lookup"><span data-stu-id="43bf4-129">The sacrifice for making SOAP easily understood is size; SOAP messages are larger than messages in comparable protocols.</span></span> <span data-ttu-id="43bf4-130">SOAP 的設計目的是將整合型應用程式分解成 multicomponent 形式，而不會失去安全性或控制。</span><span class="sxs-lookup"><span data-stu-id="43bf4-130">SOAP was designed to break monolithic applications into multicomponent form without losing security or control.</span></span> <span data-ttu-id="43bf4-131">因此，WCF 是設計用來處理這種類型的系統，而不像 gRPC，它是以分散式系統的形式開始。</span><span class="sxs-lookup"><span data-stu-id="43bf4-131">So WCF was designed to work with that kind of system, unlike gRPC, which began as a distributed system.</span></span> <span data-ttu-id="43bf4-132">WCF 藉由開發和記載 SOAP 堆疊專屬的延伸模組通訊協定來解決其中一些限制，但代價是其他平臺缺乏支援。</span><span class="sxs-lookup"><span data-stu-id="43bf4-132">WCF addressed some of these limitations by developing and documenting proprietary extension protocols for the SOAP stack, but at the cost of a lack of support from other platforms.</span></span>

<span data-ttu-id="43bf4-133">Windows Communication Foundation 是用來建立服務的架構。</span><span class="sxs-lookup"><span data-stu-id="43bf4-133">Windows Communication Foundation is a framework for building services.</span></span> <span data-ttu-id="43bf4-134">它是在早期2000s 中設計，可協助開發人員使用早期 SOA 來管理使用 SOAP 的複雜性。</span><span class="sxs-lookup"><span data-stu-id="43bf4-134">It was designed in the early 2000s to help developers using early SOA to manage the complexities of working with SOAP.</span></span> <span data-ttu-id="43bf4-135">雖然開發人員不需要撰寫自己的 SOAP 通訊協定，但 WCF 仍會使用 SOAP 來啟用與其他系統的互通性。</span><span class="sxs-lookup"><span data-stu-id="43bf4-135">Although it removes the requirement for the developers to write their own SOAP protocols, WCF still uses SOAP to enable interoperability with other systems.</span></span> <span data-ttu-id="43bf4-136">WCF 也是設計來提供跨多個通訊協定的解決方案， (HTTP/1.1、Net.tcp 等) 。</span><span class="sxs-lookup"><span data-stu-id="43bf4-136">WCF was also designed to deliver solutions across multiple protocols (HTTP/1.1, Net.TCP, and so on).</span></span>

## <a name="microservices"></a><span data-ttu-id="43bf4-137">微服務</span><span class="sxs-lookup"><span data-stu-id="43bf4-137">Microservices</span></span>

<span data-ttu-id="43bf4-138">在微服務架構中，大型的應用程式是以較小的模組化服務集合來建立。</span><span class="sxs-lookup"><span data-stu-id="43bf4-138">In microservice architectures, large applications are built as a collection of smaller modular services.</span></span> <span data-ttu-id="43bf4-139">每個元件都會執行特定的工作或程式，而元件則是設計來 interoperably 工作，但可以視需要加以隔離。</span><span class="sxs-lookup"><span data-stu-id="43bf4-139">Each component does a specific task or process, and components are designed to work interoperably but can be isolated as necessary.</span></span>

<span data-ttu-id="43bf4-140">微服務的優點包括：</span><span class="sxs-lookup"><span data-stu-id="43bf4-140">Advantages to microservices include:</span></span>

- <span data-ttu-id="43bf4-141">變更和升級可以獨立處理。</span><span class="sxs-lookup"><span data-stu-id="43bf4-141">Changes and upgrades can be handled independently.</span></span>
- <span data-ttu-id="43bf4-142">錯誤處理會變得更有效率，因為問題可以追蹤至特定服務，這些服務接著會獨立于其他服務之外隔離、重建、測試和重新部署。</span><span class="sxs-lookup"><span data-stu-id="43bf4-142">Error handling becomes more efficient because problems can be traced to specific services that are then isolated, rebuilt, tested, and redeployed independently of the other services.</span></span>
- <span data-ttu-id="43bf4-143">擴充性可局限于特定的實例或服務，而不是整個應用程式。</span><span class="sxs-lookup"><span data-stu-id="43bf4-143">Scalability can be confined to specific instances or services rather than the whole application.</span></span>
- <span data-ttu-id="43bf4-144">開發可以跨多個小組進行，相較于許多團隊在單一程式碼基底上工作時所發生的衝突。</span><span class="sxs-lookup"><span data-stu-id="43bf4-144">Development can happen across multiple teams, with less friction than occurs when many teams work on a single codebase.</span></span>

<span data-ttu-id="43bf4-145">不斷增加虛擬化、雲端運算、容器和物聯網的進展，一直微服務。</span><span class="sxs-lookup"><span data-stu-id="43bf4-145">The move towards increasing virtualization, cloud computing, containers, and the Internet of Things has contributed to the ongoing rise of microservices.</span></span> <span data-ttu-id="43bf4-146">但是微服務不會有任何挑戰。</span><span class="sxs-lookup"><span data-stu-id="43bf4-146">But microservices aren't without their challenges.</span></span> <span data-ttu-id="43bf4-147">分散/非集中式基礎結構更重視在服務之間進行通訊時所需的簡單和速度。</span><span class="sxs-lookup"><span data-stu-id="43bf4-147">The fragmented/decentralized infrastructure put more emphasis on the need for simplicity and speed when communicating between services.</span></span> <span data-ttu-id="43bf4-148">這將會特別注意 SOAP 的繁瑣和 contorted 本質。</span><span class="sxs-lookup"><span data-stu-id="43bf4-148">This in turn drew attention to the sometimes laborious and contorted nature of SOAP.</span></span>

<span data-ttu-id="43bf4-149">在此環境中，gRPC 是在 Microsoft 首次發行 WCF 後的10年推出。</span><span class="sxs-lookup"><span data-stu-id="43bf4-149">It was into this environment that gRPC was launched, 10 years after Microsoft first released WCF.</span></span> <span data-ttu-id="43bf4-150">直接從 Google 的內部基礎結構 RPC (Stubby) 發展，gRPC 永遠不會以相同的標準和通訊協定為基礎，也就是已告知許多先前 Rpc 參數的相同標準和通訊協定。</span><span class="sxs-lookup"><span data-stu-id="43bf4-150">Evolved directly from Google's internal infrastructure RPC (Stubby), gRPC was never based on the same standards and protocols that had informed the parameters of many earlier RPCs.</span></span> <span data-ttu-id="43bf4-151">而且 gRPC 只會以 HTTP/2 為基礎。</span><span class="sxs-lookup"><span data-stu-id="43bf4-151">And gRPC was only ever based on HTTP/2.</span></span> <span data-ttu-id="43bf4-152">這就是為什麼它可以繪製先進傳輸通訊協定所提供的新功能。</span><span class="sxs-lookup"><span data-stu-id="43bf4-152">That's why it could draw on the new capabilities that advanced transport protocol provided.</span></span> <span data-ttu-id="43bf4-153">尤其是雙向串流、二進位訊息和多工處理。</span><span class="sxs-lookup"><span data-stu-id="43bf4-153">In particular, bidirectional streaming, binary messaging, and multiplexing.</span></span>
