## <a name="how-an-object-ends-up-on-the-loh"></a><span data-ttu-id="8de29-111">物件如何在 LOH 上結束</span><span class="sxs-lookup"><span data-stu-id="8de29-111">How an object ends up on the LOH</span></span>

<span data-ttu-id="8de29-112">如果物件的大小大於或等於85000個位元組，就會被視為大型物件。</span><span class="sxs-lookup"><span data-stu-id="8de29-112">If an object is greater than or equal to 85,000 bytes in size, it’s considered a large object.</span></span> <span data-ttu-id="8de29-113">這個數目取決於效能微調。</span><span class="sxs-lookup"><span data-stu-id="8de29-113">This number was determined by performance tuning.</span></span> <span data-ttu-id="8de29-114">當物件配置要求是針對 85,000 個以上的位元組時，執行階段會將它配置在大型物件堆積上。</span><span class="sxs-lookup"><span data-stu-id="8de29-114">When an object allocation request is for 85,000 or more bytes, the runtime allocates it on the large object heap.</span></span>

<span data-ttu-id="8de29-115">若要瞭解這是什麼意思，檢查垃圾收集行程的一些基本概念是很有用的。</span><span class="sxs-lookup"><span data-stu-id="8de29-115">To understand what this means, it's useful to examine some fundamentals about the garbage collector.</span></span>

<span data-ttu-id="8de29-116">垃圾收集行程是一種代的收集器。</span><span class="sxs-lookup"><span data-stu-id="8de29-116">The garbage collector is a generational collector.</span></span> <span data-ttu-id="8de29-117">它有三個層代：層代 0、層代 1 和層代 2。</span><span class="sxs-lookup"><span data-stu-id="8de29-117">It has three generations: generation 0, generation 1, and generation 2.</span></span> <span data-ttu-id="8de29-118">具有 3 個層代的原因是，在精密微調的應用程式中，大部分的物件都會在 gen0 中結束。</span><span class="sxs-lookup"><span data-stu-id="8de29-118">The reason for having 3 generations is that, in a well-tuned app, most objects die in gen0.</span></span> <span data-ttu-id="8de29-119">例如，在伺服器應用程式中，與每個要求相關聯的配置應該在要求完成之後結束。</span><span class="sxs-lookup"><span data-stu-id="8de29-119">For example, in a server app, the allocations associated with each request should die after the request is finished.</span></span> <span data-ttu-id="8de29-120">執行中的配置要求則會進入 gen1，並且在該處結束。</span><span class="sxs-lookup"><span data-stu-id="8de29-120">The in-flight allocation requests will make it into gen1 and die there.</span></span> <span data-ttu-id="8de29-121">基本上，gen1 是扮演年輕物件區域與長期存留物件區域之間的緩衝區。</span><span class="sxs-lookup"><span data-stu-id="8de29-121">Essentially, gen1 acts as a buffer between young object areas and long-lived object areas.</span></span>

<span data-ttu-id="8de29-122">小物件一律會配置在層代 0 中，並根據其存留期，可能會提升至層代 1 或層代 2。</span><span class="sxs-lookup"><span data-stu-id="8de29-122">Small objects are always allocated in generation 0 and, depending on their lifetime, may be promoted to generation 1 or generation2.</span></span> <span data-ttu-id="8de29-123">大型物件則一律配置在層代 2 中。</span><span class="sxs-lookup"><span data-stu-id="8de29-123">Large objects are always allocated in generation 2.</span></span>

<span data-ttu-id="8de29-124">大型物件屬於層代 2，因為只有在層代 2 回收期間才會回收它們。</span><span class="sxs-lookup"><span data-stu-id="8de29-124">Large objects belong to generation 2 because they are collected only during a generation 2 collection.</span></span> <span data-ttu-id="8de29-125">回收層代時，也會回收其所有較年輕的層代。</span><span class="sxs-lookup"><span data-stu-id="8de29-125">When a generation is collected, all its younger generation(s) are also collected.</span></span> <span data-ttu-id="8de29-126">舉例來說，發生層代 1 GC 時，會同時回收層代 1 和 0。</span><span class="sxs-lookup"><span data-stu-id="8de29-126">For example, when a generation 1 GC happens, both generation 1 and 0 are collected.</span></span> <span data-ttu-id="8de29-127">而發生層代 2 GC 時，就會回收整個堆積。</span><span class="sxs-lookup"><span data-stu-id="8de29-127">And when a generation 2 GC happens, the whole heap is collected.</span></span> <span data-ttu-id="8de29-128">基於這個理由，層代 2 GC 也稱為「完整 GC」\*\*。</span><span class="sxs-lookup"><span data-stu-id="8de29-128">For this reason, a generation 2 GC is also called a *full GC*.</span></span> <span data-ttu-id="8de29-129">本文引用層代 2 GC，而不是完整 GC，但這些詞彙可以互換。</span><span class="sxs-lookup"><span data-stu-id="8de29-129">This article refers to generation 2 GC instead of full GC, but the terms are interchangeable.</span></span>

<span data-ttu-id="8de29-130">層代提供 GC 堆積的邏輯檢視。</span><span class="sxs-lookup"><span data-stu-id="8de29-130">Generations provide a logical view of the GC heap.</span></span> <span data-ttu-id="8de29-131">實際上，物件是存留在受控堆積區段中。</span><span class="sxs-lookup"><span data-stu-id="8de29-131">Physically, objects live in managed heap segments.</span></span> <span data-ttu-id="8de29-132">「受控堆積區段」\*\* 是記憶體的區塊，由 GC 代替受控程式碼向 OS (透過呼叫 [VirtualAlloc 函式](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc)) 要求保留。</span><span class="sxs-lookup"><span data-stu-id="8de29-132">A *managed heap segment* is a chunk of memory that the GC reserves from the OS by calling the [VirtualAlloc function](/windows/desktop/api/memoryapi/nf-memoryapi-virtualalloc) on behalf of managed code.</span></span> <span data-ttu-id="8de29-133">載入 CLR 時，GC 會配置兩個初始堆積區段：一個用於小型物件（小型物件堆積或 SOH），一個用於大型物件（大型物件堆積）。</span><span class="sxs-lookup"><span data-stu-id="8de29-133">When the CLR is loaded, the GC allocates two initial heap segments: one for small objects (the small object heap, or SOH), and one for large objects (the large object heap).</span></span>

<span data-ttu-id="8de29-134">藉由將受控物件置於這些受控堆積區段中，就可以滿足配置的要求了。</span><span class="sxs-lookup"><span data-stu-id="8de29-134">The allocation requests are then satisfied by putting managed objects on these managed heap segments.</span></span> <span data-ttu-id="8de29-135">如果物件小於 85,000 個位元組，它會置於 SOH 的區段上；否則，它會置於 LOH 區段。</span><span class="sxs-lookup"><span data-stu-id="8de29-135">If the object is less than 85,000 bytes, it is put on the segment for the SOH; otherwise, it is put on an LOH segment.</span></span> <span data-ttu-id="8de29-136">隨著愈來愈多的物件配置在區段之上，這些區段就會被認可 (以較小的區塊)。</span><span class="sxs-lookup"><span data-stu-id="8de29-136">Segments are committed (in smaller chunks) as more and more objects are allocated onto them.</span></span>
<span data-ttu-id="8de29-137">對於 SOH，在 GC 之後存留下來的物件會提升至下一個層代。</span><span class="sxs-lookup"><span data-stu-id="8de29-137">For the SOH, objects that survive a GC are promoted to the next generation.</span></span> <span data-ttu-id="8de29-138">層代 0 回收之後存留下來的物件目前被視為層代 1 物件，依此類推。</span><span class="sxs-lookup"><span data-stu-id="8de29-138">Objects that survive a generation 0 collection are now considered generation 1 objects, and so on.</span></span> <span data-ttu-id="8de29-139">然而，通過最老層代存留的物件仍可視為屬於最老的層代。</span><span class="sxs-lookup"><span data-stu-id="8de29-139">However, objects that survive the oldest generation are still considered to be in the oldest generation.</span></span> <span data-ttu-id="8de29-140">換句話說，來自層代 2 的存留者就是層代 2 物件；來自 LOH 的存留者就是 LOH 的物件 (用 gen2 來回收)。</span><span class="sxs-lookup"><span data-stu-id="8de29-140">In other words, survivors from generation 2 are generation 2 objects; and survivors from the LOH are LOH objects (which are collected with gen2).</span></span>

<span data-ttu-id="8de29-141">使用者的程式碼只能配置在層代 0 (小型物件) 或 LOH (大型物件) 中。</span><span class="sxs-lookup"><span data-stu-id="8de29-141">User code can only allocate in generation 0 (small objects) or the LOH (large objects).</span></span> <span data-ttu-id="8de29-142">唯有 GC 才能「配置」物件於層代 1 (藉由提升來自層代 0 的存留者) 與層代 2 (藉由提升來自層代 1 和 2 的存留者) 中。</span><span class="sxs-lookup"><span data-stu-id="8de29-142">Only the GC can “allocate” objects in generation 1 (by promoting survivors from generation 0) and generation 2 (by promoting survivors from generations 1 and 2).</span></span>

<span data-ttu-id="8de29-143">觸發記憶體回收時，GC 會追蹤所有作用中的物件，並且加以壓縮。</span><span class="sxs-lookup"><span data-stu-id="8de29-143">When a garbage collection is triggered, the GC traces through the live objects and compacts them.</span></span> <span data-ttu-id="8de29-144">但因為壓縮要耗費大量資源，所以 GC 會「掃除」\*\* LOH；這可將無作用物件建立成一份可用清單，稍後再用此清單來滿足大型物件配置的要求。</span><span class="sxs-lookup"><span data-stu-id="8de29-144">But because compaction is expensive, the GC *sweeps* the LOH; it makes a free list out of dead objects that can be reused later to satisfy large object allocation requests.</span></span> <span data-ttu-id="8de29-145">鄰近的無作用物件會結合成為一個可用物件。</span><span class="sxs-lookup"><span data-stu-id="8de29-145">Adjacent dead objects are made into one free object.</span></span>

## <a name="when-is-a-large-object-collected"></a><span data-ttu-id="8de29-169">何時收集大型物件？</span><span class="sxs-lookup"><span data-stu-id="8de29-169">When is a large object collected?</span></span>

## <a name="loh-performance-implications"></a><span data-ttu-id="8de29-184">LOH 效能的影響</span><span class="sxs-lookup"><span data-stu-id="8de29-184">LOH performance implications</span></span>

- <span data-ttu-id="8de29-192">回收成本。</span><span class="sxs-lookup"><span data-stu-id="8de29-192">Collection cost.</span></span>
