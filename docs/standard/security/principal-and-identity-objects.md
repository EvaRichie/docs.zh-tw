## <a name="identity-objects"></a><span data-ttu-id="ef608-113">Identity 物件</span><span class="sxs-lookup"><span data-stu-id="ef608-113">Identity Objects</span></span>

<span data-ttu-id="ef608-114">Identity 物件會封裝已驗證的使用者或實體相關資訊。</span><span class="sxs-lookup"><span data-stu-id="ef608-114">The identity object encapsulates information about the user or entity being validated.</span></span> <span data-ttu-id="ef608-115">在其最基本層級中，Identity 物件會包含名稱和驗證類型。</span><span class="sxs-lookup"><span data-stu-id="ef608-115">At their most basic level, identity objects contain a name and an authentication type.</span></span> <span data-ttu-id="ef608-116">名稱可以是使用者的名稱或 Windows 帳戶的名稱，而驗證類型可以是支援的登入通訊協定，例如 Kerberos V5 或自訂值。</span><span class="sxs-lookup"><span data-stu-id="ef608-116">The name can either be a user's name or the name of a Windows account, while the authentication type can be either a supported logon protocol, such as Kerberos V5, or a custom value.</span></span> <span data-ttu-id="ef608-117">.NET 會定義一個 <xref:System.Security.Principal.GenericIdentity> 物件，可用於大部分的自訂登入案例，以及 <xref:System.Security.Principal.WindowsIdentity> 當您想要讓應用程式依賴 Windows 驗證時，可使用更特製化的物件。</span><span class="sxs-lookup"><span data-stu-id="ef608-117">.NET defines a <xref:System.Security.Principal.GenericIdentity> object that can be used for most custom logon scenarios and a more specialized <xref:System.Security.Principal.WindowsIdentity> object that can be used when you want your application to rely on Windows authentication.</span></span> <span data-ttu-id="ef608-118">此外，您可以定義自己的身分識別類別，它會封裝自訂使用者資訊。</span><span class="sxs-lookup"><span data-stu-id="ef608-118">Additionally, you can define your own identity class that encapsulates custom user information.</span></span>  
  
<span data-ttu-id="ef608-119"><xref:System.Security.Principal.IIdentity>介面會定義用來存取名稱和驗證類型（例如 Kerberos V5 或 NTLM）的屬性。</span><span class="sxs-lookup"><span data-stu-id="ef608-119">The <xref:System.Security.Principal.IIdentity> interface defines properties for accessing a name and an authentication type, such as Kerberos V5 or NTLM.</span></span> <span data-ttu-id="ef608-120">所有 **身分識別** 類別會實作 **IIdentity** 介面。</span><span class="sxs-lookup"><span data-stu-id="ef608-120">All **Identity** classes implement the **IIdentity** interface.</span></span> <span data-ttu-id="ef608-121">**Identity** 物件和 Windows NT 處理序語彙基元 (執行緒目前在其底下執行) 之間沒有任何必要的關聯性。</span><span class="sxs-lookup"><span data-stu-id="ef608-121">There is no required relationship between an **Identity** object and the Windows NT process token under which a thread is currently executing.</span></span> <span data-ttu-id="ef608-122">不過，如果 **Identity** 物件是 **WindowsIdentity** 物件時，身分識別會假設為代表 Windows NT 安全性語彙基元。</span><span class="sxs-lookup"><span data-stu-id="ef608-122">However, if the **Identity** object is a **WindowsIdentity** object, the identity is assumed to represent a Windows NT security token.</span></span>  
  
## <a name="principal-objects"></a><span data-ttu-id="ef608-123">Principal 物件</span><span class="sxs-lookup"><span data-stu-id="ef608-123">Principal Objects</span></span>

<span data-ttu-id="ef608-124">Principal 物件代表執行程式碼所在的安全性內容。</span><span class="sxs-lookup"><span data-stu-id="ef608-124">The principal object represents the security context under which code is running.</span></span> <span data-ttu-id="ef608-125">實作以角色為基礎的安全性之應用程式，會根據與 Principal 物件相關聯的角色授與權限。</span><span class="sxs-lookup"><span data-stu-id="ef608-125">Applications that implement role-based security grant rights based on the role associated with a principal object.</span></span> <span data-ttu-id="ef608-126">與 identity 物件類似，.NET 提供了 <xref:System.Security.Principal.GenericPrincipal> 物件和 <xref:System.Security.Principal.WindowsPrincipal> 物件。</span><span class="sxs-lookup"><span data-stu-id="ef608-126">Similar to identity objects, .NET provides a <xref:System.Security.Principal.GenericPrincipal> object and a <xref:System.Security.Principal.WindowsPrincipal> object.</span></span> <span data-ttu-id="ef608-127">您也可以定義自己的自訂主體類別。</span><span class="sxs-lookup"><span data-stu-id="ef608-127">You can also define your own custom principal classes.</span></span>  
  
<span data-ttu-id="ef608-128"><xref:System.Security.Principal.IPrincipal>介面會定義用來存取相關聯身分 **識別** 物件的屬性，以及用來判斷 **主體** 物件所識別的使用者是否為指定角色成員的方法。</span><span class="sxs-lookup"><span data-stu-id="ef608-128">The <xref:System.Security.Principal.IPrincipal> interface defines a property for accessing an associated **Identity** object as well as a method for determining whether the user identified by the **Principal** object is a member of a given role.</span></span> <span data-ttu-id="ef608-129">所有 **Principal** 類別會實作 **IPrincipal** 介面以及任何其他需要的屬性和方法。</span><span class="sxs-lookup"><span data-stu-id="ef608-129">All **Principal** classes implement the **IPrincipal** interface as well as any additional properties and methods that are necessary.</span></span> <span data-ttu-id="ef608-130">例如，通用語言執行平台提供 **WindowsPrincipal** 類別，實作其他功能以將 Windows NT 或 Windows 2000 群組成員資格對應至角色。</span><span class="sxs-lookup"><span data-stu-id="ef608-130">For example, the common language runtime provides the **WindowsPrincipal** class, which implements additional functionality for mapping Windows NT or Windows 2000 group membership to roles.</span></span>  
  
<span data-ttu-id="ef608-131">**主體** 物件會系結至 <xref:System.Runtime.Remoting.Messaging.CallContext> 應用程式域中的呼叫內容 () 物件 (<xref:System.AppDomain>) 。</span><span class="sxs-lookup"><span data-stu-id="ef608-131">A **Principal** object is bound to a call context (<xref:System.Runtime.Remoting.Messaging.CallContext>) object within an application domain (<xref:System.AppDomain>).</span></span> <span data-ttu-id="ef608-132">預設呼叫內容一律使用每個新的 **AppDomain** 建立，因此永遠有呼叫內容可以接受 **Principal** 物件。</span><span class="sxs-lookup"><span data-stu-id="ef608-132">A default call context is always created with each new **AppDomain**, so there is always a call context available to accept the **Principal** object.</span></span> <span data-ttu-id="ef608-133">建立新的執行緒時，也會針對執行緒建立 **CallContext** 物件。</span><span class="sxs-lookup"><span data-stu-id="ef608-133">When a new thread is created, a **CallContext** object is also created for the thread.</span></span> <span data-ttu-id="ef608-134">**Principal** 物件參考會自動從建立的執行緒複製到新執行緒的 **CallContext**。</span><span class="sxs-lookup"><span data-stu-id="ef608-134">The **Principal** object reference is automatically copied from the creating thread to the new thread's **CallContext**.</span></span> <span data-ttu-id="ef608-135">如果執行階段無法判斷哪個 **Principal** 物件屬於執行緒的建立者，它會依照 **Principal** 和 **Identity** 物件建立的預設原則。</span><span class="sxs-lookup"><span data-stu-id="ef608-135">If the runtime cannot determine which **Principal** object belongs to the creator of the thread, it follows the default policy for **Principal** and **Identity** object creation.</span></span>  
  
<span data-ttu-id="ef608-136">可設定的應用程式定義域特定原則會定義規則，來決定哪種類型的 **Principal** 物件與新的應用程式定義域相關聯。</span><span class="sxs-lookup"><span data-stu-id="ef608-136">A configurable application domain-specific policy defines the rules for deciding what type of **Principal** object to associate with a new application domain.</span></span> <span data-ttu-id="ef608-137">安全性原則允許時，執行階段可以建立 **Principal** 和 **Identity** 物件，反映與目前執行緒相關聯的作業系統語彙基元。</span><span class="sxs-lookup"><span data-stu-id="ef608-137">Where security policy permits, the runtime can create **Principal** and **Identity** objects that reflect the operating system token associated with the current thread of execution.</span></span> <span data-ttu-id="ef608-138">根據預設，執行階段會使用 **Principal** 和 **Identity** 物件，代表未經授權的使用者。</span><span class="sxs-lookup"><span data-stu-id="ef608-138">By default, the runtime uses **Principal** and **Identity** objects that represent unauthenticated users.</span></span> <span data-ttu-id="ef608-139">執行階段不會建立這些預設 **Principal** 和 **Identity** 物件，直到程式碼嘗試存取它們為止。</span><span class="sxs-lookup"><span data-stu-id="ef608-139">The runtime does not create these default **Principal** and **Identity** objects until the code attempts to access them.</span></span>  
  
<span data-ttu-id="ef608-140">會建立應用程式定義域之信任的程式碼，可以設定應用程式定義域原則，控制預設 **Principal** 和 **Identity** 物件的建構。</span><span class="sxs-lookup"><span data-stu-id="ef608-140">Trusted code that creates an application domain can set the application domain policy that controls construction of the default **Principal** and **Identity** objects.</span></span> <span data-ttu-id="ef608-141">此應用程式定義域專屬的原則適用於所有應用程式定義域中的執行緒。</span><span class="sxs-lookup"><span data-stu-id="ef608-141">This application domain-specific policy applies to all execution threads in that application domain.</span></span> <span data-ttu-id="ef608-142">非受控、受信任的主機原本就可以設定此原則，但設定此原則的 managed 程式碼必須具有 <xref:System.Security.Permissions.SecurityPermission?displayProperty=nameWithType> 控制網域原則的。</span><span class="sxs-lookup"><span data-stu-id="ef608-142">An unmanaged, trusted host inherently has the ability to set this policy, but managed code that sets this policy must have the <xref:System.Security.Permissions.SecurityPermission?displayProperty=nameWithType> for controlling domain policy.</span></span>  
  
<span data-ttu-id="ef608-143">跨應用程式定義域但是在相同處理序內傳輸 **Principal** 物件時 (因此是在同一部電腦上)，遠端基礎結構會將參考複製到 **Principal** 物件，該物件將呼叫者的內容與被呼叫端的內容產生關聯。</span><span class="sxs-lookup"><span data-stu-id="ef608-143">When transmitting a **Principal** object across application domains but within the same process (and therefore on the same computer), the remoting infrastructure copies a reference to the **Principal** object associated with the caller's context to the callee's context.</span></span>
