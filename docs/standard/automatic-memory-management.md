## <a name="generations-and-performance"></a><span data-ttu-id="040ce-140">層代和效能</span><span class="sxs-lookup"><span data-stu-id="040ce-140">Generations and Performance</span></span>  
 <span data-ttu-id="040ce-141">為了要最佳化記憶體回收行程的效能，Managed 堆積分成三個層代 (Generation)：0、1 和 2。</span><span class="sxs-lookup"><span data-stu-id="040ce-141">To optimize the performance of the garbage collector, the managed heap is divided into three generations: 0, 1, and 2.</span></span> <span data-ttu-id="040ce-142">執行階段的記憶體回收演算法，是根據電腦軟體產業利用實驗記憶體回收配置所發現的數個通用原理。</span><span class="sxs-lookup"><span data-stu-id="040ce-142">The runtime's garbage collection algorithm is based on several generalizations that the computer software industry has discovered to be true by experimenting with garbage collection schemes.</span></span> <span data-ttu-id="040ce-143">首先，壓縮部分 Managed 堆積的記憶體比壓縮整個 Managed 堆積要快。</span><span class="sxs-lookup"><span data-stu-id="040ce-143">First, it is faster to compact the memory for a portion of the managed heap than for the entire managed heap.</span></span> <span data-ttu-id="040ce-144">其次，較新物件的存留期 (Lifetime) 較短，較舊物件的存留期較長。</span><span class="sxs-lookup"><span data-stu-id="040ce-144">Secondly, newer objects will have shorter lifetimes and older objects will have longer lifetimes.</span></span> <span data-ttu-id="040ce-145">最後，較新的物件通常都會彼此相關，而且應用程式也會同時存取。</span><span class="sxs-lookup"><span data-stu-id="040ce-145">Lastly, newer objects tend to be related to each other and accessed by the application around the same time.</span></span>  
  
 <span data-ttu-id="040ce-146">執行階段的記憶體回收行程會將新的物件儲存在第 0 個層代。</span><span class="sxs-lookup"><span data-stu-id="040ce-146">The runtime's garbage collector stores new objects in generation 0.</span></span> <span data-ttu-id="040ce-147">在應用程式存留期初期建立的物件則會升階並儲存在第 1 個和第 2 個層代。</span><span class="sxs-lookup"><span data-stu-id="040ce-147">Objects created early in the application's lifetime that survive collections are promoted and stored in generations 1 and 2.</span></span> <span data-ttu-id="040ce-148">物件提升的程序會在這個主題稍後加以說明。</span><span class="sxs-lookup"><span data-stu-id="040ce-148">The process of object promotion is described later in this topic.</span></span> <span data-ttu-id="040ce-149">由於壓縮部分 Managed 堆積比壓縮整個堆積要快，因此這種配置允許記憶體回收行程釋放指定層代的記憶體，而不是在每次執行回收時釋放整個 Managed 堆積的記憶體。</span><span class="sxs-lookup"><span data-stu-id="040ce-149">Because it is faster to compact a portion of the managed heap than the entire heap, this scheme allows the garbage collector to release the memory in a specific generation rather than release the memory for the entire managed heap each time it performs a collection.</span></span>  
  
 <span data-ttu-id="040ce-150">事實上，當第 0 個層代已滿時，記憶體回收行程便會執行回收。</span><span class="sxs-lookup"><span data-stu-id="040ce-150">In reality, the garbage collector performs a collection when generation 0 is full.</span></span> <span data-ttu-id="040ce-151">當第 0 個層代已滿時，如果應用程式嘗試建立新的物件，記憶體回收行程會發現第 0 個層代中已經沒有剩餘的位址空間可配置給物件。</span><span class="sxs-lookup"><span data-stu-id="040ce-151">If an application attempts to create a new object when generation 0 is full, the garbage collector discovers that there is no address space remaining in generation 0 to allocate for the object.</span></span> <span data-ttu-id="040ce-152">這時記憶體回收行程便會執行回收，嘗試為該物件釋放出第 0 個層代的位址空間。</span><span class="sxs-lookup"><span data-stu-id="040ce-152">The garbage collector performs a collection in an attempt to free address space in generation 0 for the object.</span></span> <span data-ttu-id="040ce-153">記憶體回收行程是從檢查第 0 個層代中的物件開始，而不是檢查 Managed 堆積中的所有物件。</span><span class="sxs-lookup"><span data-stu-id="040ce-153">The garbage collector starts by examining the objects in generation 0 rather than all objects in the managed heap.</span></span> <span data-ttu-id="040ce-154">這是最有效率的方式，因為新物件的存留期通常較短，因此在執行回收時，通常會認為應用程式已不再使用第 0 個層代中的許多物件。</span><span class="sxs-lookup"><span data-stu-id="040ce-154">This is the most efficient approach, because new objects tend to have short lifetimes, and it is expected that many of the objects in generation 0 will no longer be in use by the application when a collection is performed.</span></span> <span data-ttu-id="040ce-155">此外，單獨回收第 0 個層代通常就能收回足夠的記憶體，讓應用程式繼續建立新的物件。</span><span class="sxs-lookup"><span data-stu-id="040ce-155">In addition, a collection of generation 0 alone often reclaims enough memory to allow the application to continue creating new objects.</span></span>  
  
 <span data-ttu-id="040ce-156">在記憶體回收行程執行層代 0 的回收之後，會壓縮可執行物件的記憶體，如同本主題中前面的[釋放記憶體](#cpconautomaticmemorymanagementreleasingmemoryanchor1)中的說明。</span><span class="sxs-lookup"><span data-stu-id="040ce-156">After the garbage collector performs a collection of generation 0, it compacts the memory for the reachable objects as explained in [Releasing Memory](#cpconautomaticmemorymanagementreleasingmemoryanchor1) earlier in this topic.</span></span> <span data-ttu-id="040ce-157">然後，記憶體回收行程會提升這些物件，並考慮 Managed 堆積第 1 個層代的這個部分。</span><span class="sxs-lookup"><span data-stu-id="040ce-157">The garbage collector then promotes these objects and considers this portion of the managed heap generation 1.</span></span> <span data-ttu-id="040ce-158">由於回收之後存留下來的物件通常具有較長的存留期，因此才會將它們提升至較高的層代。</span><span class="sxs-lookup"><span data-stu-id="040ce-158">Because objects that survive collections tend to have longer lifetimes, it makes sense to promote them to a higher generation.</span></span> <span data-ttu-id="040ce-159">如此一來，每當記憶體回收行程執行第 0 個層代的回收時，就不需要再重新檢查第 1 和第 2 個層代中的物件了。</span><span class="sxs-lookup"><span data-stu-id="040ce-159">As a result, the garbage collector does not have to reexamine the objects in generations 1 and 2 each time it performs a collection of generation 0.</span></span>

data-ttu-id="040ce-161">接著會繼續為新的物件配置第 0 個層代中的記憶體，直到第 0 個層代已滿而必須再執行另一次回收為止。</span><span class="sxs-lookup"><span data-stu-id="040ce-161">It continues to allocate memory for new objects in generation 0 generation 2.</span></span> <span data-ttu-id="040ce-164">如果仍然沒有收回足夠的記憶體，記憶體回收行程可以再依序執行第 2、第 1 和第 0 個層代的回收。</span><span class="sxs-lookup"><span data-stu-id="040ce-164">If this does not reclaim enough
