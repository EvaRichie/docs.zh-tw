# <a name="runtime-callable-wrapper"></a><span data-ttu-id="17ed1-103">執行階段可呼叫包裝函式</span><span class="sxs-lookup"><span data-stu-id="17ed1-103">Runtime Callable Wrapper</span></span>
<span data-ttu-id="17ed1-104">Common Language Runtime 會透過名為執行階段可呼叫包裝函式 (RCW) 的 Proxy 來公開 COM 物件。</span><span class="sxs-lookup"><span data-stu-id="17ed1-104">The common language runtime exposes COM objects through a proxy called the runtime callable wrapper (RCW).</span></span> <span data-ttu-id="17ed1-105">雖然 RCW 對 .NET 用戶端似乎是個普通物件，但其主要功能是在 .NET 用戶端與 COM 物件之間封送處理呼叫。</span><span class="sxs-lookup"><span data-stu-id="17ed1-105">Although the RCW appears to be an ordinary object to .NET clients, its primary function is to marshal calls between a .NET client and a COM object.</span></span>  
  
 <span data-ttu-id="17ed1-106">不論 COM 物件上存在著多少參考，執行階段都會針對每個 COM 物件各建立一個 RCW。</span><span class="sxs-lookup"><span data-stu-id="17ed1-106">The runtime creates exactly one RCW for each COM object, regardless of the number of references that exist on that object.</span></span> <span data-ttu-id="17ed1-107">執行階段會針對每個物件的每個處理序各維護一個 RCW。</span><span class="sxs-lookup"><span data-stu-id="17ed1-107">The runtime maintains a single RCW per process for each object.</span></span>  <span data-ttu-id="17ed1-108">如果您在一個應用程式定義域或 Apartment 中建立 RCW，然後將參考傳遞至另一個應用程式定義域或 Apartment，將會使用第一個物件的 Proxy。</span><span class="sxs-lookup"><span data-stu-id="17ed1-108">If you create an RCW in one application domain or apartment, and then pass a reference to another application domain or apartment, a proxy to the first object will be used.</span></span>  <span data-ttu-id="17ed1-109">如下圖所示，任何數目的 Managed 用戶端都可以保存公開 INew 和 INewer 介面之 COM 物件的參考。</span><span class="sxs-lookup"><span data-stu-id="17ed1-109">As the following illustration shows, any number of managed clients can hold a reference to the COM objects that expose INew and INewer interfaces.</span></span>  

<span data-ttu-id="17ed1-110">下圖顯示透過執行階段可呼叫包裝函式來存取 COM 物件的程序：</span><span class="sxs-lookup"><span data-stu-id="17ed1-110">The following image shows the process for accessing COM objects through the runtime callable wrapper:</span></span>

 ![透過 RCW 存取 COM 物件的程序。](./media/runtime-callable-wrapper/runtime-callable-wrapper.gif)  

 <span data-ttu-id="17ed1-112">使用衍生自類型程式庫的中繼資料，執行階段可以建立所呼叫的 COM 物件，以及該物件的包裝函式。</span><span class="sxs-lookup"><span data-stu-id="17ed1-112">Using metadata derived from a type library, the runtime creates both the COM object being called and a wrapper for that object.</span></span> <span data-ttu-id="17ed1-113">每個 RCW 都會在其包裝的 COM 物件上維護介面指標的快取，當不再需要 RCW 時，就會在 COM 物件上釋出其參考。</span><span class="sxs-lookup"><span data-stu-id="17ed1-113">Each RCW maintains a cache of interface pointers on the COM object it wraps and releases its reference on the COM object when the RCW is no longer needed.</span></span> <span data-ttu-id="17ed1-114">執行階段會在 RCW 上執行記憶體回收。</span><span class="sxs-lookup"><span data-stu-id="17ed1-114">The runtime performs garbage collection on the RCW.</span></span>  
  
 <span data-ttu-id="17ed1-115">RCW 有許多活動，其中包括代表所包裝的物件，在 Managed 與 Unmanaged 程式碼之間封送處理資料。</span><span class="sxs-lookup"><span data-stu-id="17ed1-115">Among other activities, the RCW marshals data between managed and unmanaged code, on behalf of the wrapped object.</span></span> <span data-ttu-id="17ed1-116">具體來說，每當用戶端與伺服器之間，有不同表示方式的資料要傳遞時，RCW 就會針對方法引數和方法傳回值提供封送處理。</span><span class="sxs-lookup"><span data-stu-id="17ed1-116">Specifically, the RCW provides marshaling for method arguments and method return values whenever the client and server have different representations of the data passed between them.</span></span>  
  
 <span data-ttu-id="17ed1-117">標準包裝函式會強制執行內建的封送處理規則。</span><span class="sxs-lookup"><span data-stu-id="17ed1-117">The standard wrapper enforces built-in marshaling rules.</span></span> <span data-ttu-id="17ed1-118">例如，當 .NET 用戶端將字串類型當做引數的一部分傳遞至 Unmanaged 物件時，包裝函式會將字串轉換成 BSTR 類型。</span><span class="sxs-lookup"><span data-stu-id="17ed1-118">For example, when a .NET client passes a String type as part of an argument to an unmanaged object, the wrapper converts the string to a BSTR type.</span></span> <span data-ttu-id="17ed1-119">如果 COM 物件將 BSTR 傳回至其 Managed 呼叫端，該呼叫端會收到一個字串。</span><span class="sxs-lookup"><span data-stu-id="17ed1-119">Should the COM object return a BSTR to its managed caller, the caller receives a String.</span></span> <span data-ttu-id="17ed1-120">用戶端和伺服器都會傳送及接收其熟悉的資料。</span><span class="sxs-lookup"><span data-stu-id="17ed1-120">Both the client and the server send and receive data that is familiar to them.</span></span> <span data-ttu-id="17ed1-121">其他類型則不需轉換。</span><span class="sxs-lookup"><span data-stu-id="17ed1-121">Other types require no conversion.</span></span> <span data-ttu-id="17ed1-122">比方說，標準包裝函式一律會在 Managed 與 Unmanaged 程式碼之間傳遞 4 位元組的整數，而不會轉換類型。</span><span class="sxs-lookup"><span data-stu-id="17ed1-122">For instance, a standard wrapper will always pass a 4-byte integer between managed and unmanaged code without converting the type.</span></span>
