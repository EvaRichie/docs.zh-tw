# <a name="multiple-endpoints-at-a-single-listenuri"></a><span data-ttu-id="cac74-102">單一 ListenUri 的多個端點</span><span class="sxs-lookup"><span data-stu-id="cac74-102">Multiple Endpoints at a Single ListenUri</span></span>

> [!NOTE]
> <span data-ttu-id="cac74-105">此範例的安裝程序與建置指示位於本主題的結尾。</span><span class="sxs-lookup"><span data-stu-id="cac74-105">The setup procedure and build instructions for this sample are located at the end of this topic.</span></span>  
  
 <span data-ttu-id="cac74-106">如 [多個端點](multiple-endpoints.md) 範例所示，服務可以裝載多個端點，每個端點都有不同的位址，而且可能也有不同的系結。</span><span class="sxs-lookup"><span data-stu-id="cac74-106">As demonstrated in the [Multiple Endpoints](multiple-endpoints.md) sample, a service can host multiple endpoints, each with different addresses and possibly also different bindings.</span></span> <span data-ttu-id="cac74-107">這個範例會示範如何在相同位址裝載多個端點。</span><span class="sxs-lookup"><span data-stu-id="cac74-107">This sample shows that it is possible to host multiple endpoints at the same address.</span></span> <span data-ttu-id="cac74-108">此外，這個範例還會示範服務端點所具有兩種位址的差異：`EndpointAddress` 和 `ListenUri`。</span><span class="sxs-lookup"><span data-stu-id="cac74-108">This sample also demonstrates the differences between the two kinds of addresses that a service endpoint has: `EndpointAddress` and `ListenUri`.</span></span>  
  
 <span data-ttu-id="cac74-109">`EndpointAddress` 是服務的邏輯位址。</span><span class="sxs-lookup"><span data-stu-id="cac74-109">The `EndpointAddress` is the logical address of a service.</span></span> <span data-ttu-id="cac74-110">這是 SOAP 訊息傳送至的目標位址。</span><span class="sxs-lookup"><span data-stu-id="cac74-110">It is the address that SOAP messages are addressed to.</span></span> <span data-ttu-id="cac74-111">`ListenUri` 是服務的實際位址，</span><span class="sxs-lookup"><span data-stu-id="cac74-111">The `ListenUri` is the physical address of the service.</span></span> <span data-ttu-id="cac74-112">含有連接埠和位址資訊，而服務端點實際上會接聽目前電腦上的訊息。</span><span class="sxs-lookup"><span data-stu-id="cac74-112">It has the port and address information where the service endpoint actually listens for messages on the current machine.</span></span> <span data-ttu-id="cac74-113">在大多數情況下，這些位址不需要有所不同，因為未明確指定 `ListenUri` 時，預設會是端點之 `EndpointAddress` 的 URI。</span><span class="sxs-lookup"><span data-stu-id="cac74-113">In most cases, there is no need for these addresses to differ; when a `ListenUri` is not explicitly specified, it defaults to the URI of the `EndpointAddress` of the endpoint.</span></span> <span data-ttu-id="cac74-114">在少數情況下，例如設定路由器時區分這些位址就很有用，該路由器可能會接受傳送至一些不同服務的訊息。</span><span class="sxs-lookup"><span data-stu-id="cac74-114">In a few cases, it is useful to distinguish them, such as when configuring a router, which might accept messages addressed to a number of different services.</span></span>  
  
## <a name="service"></a><span data-ttu-id="cac74-115">服務</span><span class="sxs-lookup"><span data-stu-id="cac74-115">Service</span></span>  

 <span data-ttu-id="cac74-116">這個範例中的服務有兩個合約，`ICalculator` 和 `IEcho`。</span><span class="sxs-lookup"><span data-stu-id="cac74-116">The service in this sample has two contracts, `ICalculator` and `IEcho`.</span></span> <span data-ttu-id="cac74-117">除了可自訂的 `IMetadataExchange` 端點外，還有三個應用程式端點，如下列程式碼所示。</span><span class="sxs-lookup"><span data-stu-id="cac74-117">In addition to the customary `IMetadataExchange` endpoint, there are three application endpoints, as shown in the following code.</span></span>  
  
```xml  
<endpoint address="urn:Stuff"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.ICalculator"
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:Stuff"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
<endpoint address="urn:OtherEcho"  
        binding="wsHttpBinding"  
        contract="Microsoft.ServiceModel.Samples.IEcho"
        listenUri="http://localhost/servicemodelsamples/service.svc" />  
```  
  
 <span data-ttu-id="cac74-118">這三個端點都會裝載在相同的 `ListenUri`，並使用相同的 `binding`。相同 `ListenUri` 上的端點必須具有相同的繫結，這樣才可以共用接聽電腦上該實際位址之訊息的單一通道堆疊。</span><span class="sxs-lookup"><span data-stu-id="cac74-118">All three endpoints are hosted at the same `ListenUri` and use the same `binding` - endpoints at the same `ListenUri` must have the same binding, because they are sharing a single channel stack that listens for messages at that physical address on the machine.</span></span> <span data-ttu-id="cac74-119">每個端點的 `address` 為 URN。雖然一般來說，位址表示實際位置，事實上位址可以是任何種類的 URI，因為位址的用途就是比對和篩選，如本範例所示。</span><span class="sxs-lookup"><span data-stu-id="cac74-119">The `address` of each endpoint is a URN; though typically addresses represent physical locations, in fact the address can be any kind of URI, because the address is used for matching and filtering purposes as is demonstrated in this sample.</span></span>  
  
 <span data-ttu-id="cac74-120">由於這三個端點共用相同 `ListenUri` ，當訊息抵達該處時，Windows Communication Foundation (WCF) 必須決定訊息的目的地端點。</span><span class="sxs-lookup"><span data-stu-id="cac74-120">Because all three endpoints share the same `ListenUri`, when a message arrives there, Windows Communication Foundation (WCF) must decide which endpoint the message is destined for.</span></span> <span data-ttu-id="cac74-121">每個端點都具有由兩個部分組成的訊息篩選條件：位址篩選條件和合約篩選條件。</span><span class="sxs-lookup"><span data-stu-id="cac74-121">Each endpoint has a message filter that is comprised of two parts: the address filter and the contract filter.</span></span> <span data-ttu-id="cac74-122">位址篩選條件會比對 SOAP 訊息的 `To` 和服務端點的位址。</span><span class="sxs-lookup"><span data-stu-id="cac74-122">The address filter matches the `To` of the SOAP message to the address of the service endpoint.</span></span> <span data-ttu-id="cac74-123">例如，只有指定 `To "Urn:OtherEcho"` 的訊息是這個服務第三個端點的候選。</span><span class="sxs-lookup"><span data-stu-id="cac74-123">For example, only messages addressed `To "Urn:OtherEcho"` are candidates for the third endpoint of this service.</span></span> <span data-ttu-id="cac74-124">合約篩選條件會比對與特定合約作業相關聯的動作。</span><span class="sxs-lookup"><span data-stu-id="cac74-124">The contract filter matches the Actions associated with the operations of a particular contract.</span></span> <span data-ttu-id="cac74-125">例如，具有 `IEcho` 動作的訊息。</span><span class="sxs-lookup"><span data-stu-id="cac74-125">For example, messages with the action of `IEcho`.</span></span> <span data-ttu-id="cac74-126">`Echo` 會比對這個服務第二和第三個端點的合約篩選條件，因為這兩個端點都會裝載 `IEcho` 合約。</span><span class="sxs-lookup"><span data-stu-id="cac74-126">`Echo` matches the contract filters of both the second and third endpoints of this service, because both of those endpoints host the `IEcho` contract.</span></span>  
  
 <span data-ttu-id="cac74-127">因此，結合位址篩選條件和合約篩選條件之後，就可以將抵達這個服務之 `ListenUri` 的每個訊息路由至正確的端點。</span><span class="sxs-lookup"><span data-stu-id="cac74-127">Thus the combination of address filter and contract filter makes it possible to route each message that arrives at this service's `ListenUri` to the correct endpoint.</span></span> <span data-ttu-id="cac74-128">第三個端點與其他兩個端點不同，它會接受從其他端點傳送至不同位址的訊息。</span><span class="sxs-lookup"><span data-stu-id="cac74-128">The third endpoint is differentiated from the other two because it accepts messages sent to a different address from the other endpoints.</span></span> <span data-ttu-id="cac74-129">第一個和第二個端點可以依合約 (傳入訊息的動作) 區分。</span><span class="sxs-lookup"><span data-stu-id="cac74-129">The first and second endpoints are differentiated from each other based on their contracts (the Action of the incoming message).</span></span>  
  
## <a name="client"></a><span data-ttu-id="cac74-130">用戶端</span><span class="sxs-lookup"><span data-stu-id="cac74-130">Client</span></span>  

 <span data-ttu-id="cac74-131">由於伺服器上的端點有兩個不同的位址，用戶端端點也會有兩個位址。</span><span class="sxs-lookup"><span data-stu-id="cac74-131">Just as endpoints on the server have two different addresses, client endpoints also have two addresses.</span></span> <span data-ttu-id="cac74-132">在伺服器和用戶端上，邏輯位址稱為 `EndpointAddress`。</span><span class="sxs-lookup"><span data-stu-id="cac74-132">On both server and client, the logical address is called the `EndpointAddress`.</span></span> <span data-ttu-id="cac74-133">不過，在伺服器上，實際位址稱為 `ListenUri`，在用戶端上，實際位址則稱為 `Via`。</span><span class="sxs-lookup"><span data-stu-id="cac74-133">But whereas the physical address is called the `ListenUri` on the server, on the client, the physical address is called the `Via`.</span></span>  
  
 <span data-ttu-id="cac74-134">根據預設，伺服器上的這兩個位址是相同的。</span><span class="sxs-lookup"><span data-stu-id="cac74-134">As on the server, by default, these two addresses are the same.</span></span> <span data-ttu-id="cac74-135">若要在用戶端上指定與端點位址不同的 `Via`，請使用 `ClientViaBehavior`：</span><span class="sxs-lookup"><span data-stu-id="cac74-135">To specify a `Via` on the client that is different from the endpoint's address, `ClientViaBehavior` is used:</span></span>  
  
```csharp  
Uri via = new Uri("http://localhost/ServiceModelSamples/service.svc");  
CalculatorClient calcClient = new CalculatorClient();  
calcClient.ChannelFactory.Endpoint.Behaviors.Add(  
        new ClientViaBehavior(via));  
```  
  
 <span data-ttu-id="cac74-136">位址照例是來自 Svcutil.exe 所產生的用戶端組態檔。</span><span class="sxs-lookup"><span data-stu-id="cac74-136">As usual, the address comes from the client configuration file, which was generated by Svcutil.exe.</span></span> <span data-ttu-id="cac74-137">`Via` (對應至服務的 `ListenUri`) 不會顯示在服務的中繼資料中，因此這個資訊必須傳送到超出範圍的用戶端 (就和服務的中繼資料位址一樣)。</span><span class="sxs-lookup"><span data-stu-id="cac74-137">The `Via` (which corresponds to the `ListenUri` of the service) does not appear in the service's metadata and so this information must be communicated to the client out-of-band (just like the service's metadata address).</span></span>  
  
 <span data-ttu-id="cac74-138">這個範例中的用戶端會將訊息傳送至伺服器的三個應用程式端點，示範它可以與這三個端點通訊 (也可以區別這三個端點)，即使這三個端點具有相同的 `Via` 也是一樣。</span><span class="sxs-lookup"><span data-stu-id="cac74-138">The client in this sample sends messages to each of the server's three application endpoints, to demonstrate that it can communicate with (and differentiate) all three endpoints, even though they all have the same `Via`.</span></span>
