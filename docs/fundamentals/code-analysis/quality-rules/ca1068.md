## <a name="rule-description"></a><span data-ttu-id="c278f-114">規則描述</span><span class="sxs-lookup"><span data-stu-id="c278f-114">Rule description</span></span>

<span data-ttu-id="c278f-115">執行長時間執行作業或非同步作業且可取消的方法通常會使用解除標記參數。</span><span class="sxs-lookup"><span data-stu-id="c278f-115">Methods that perform long running operations or asynchronous operations and are cancelable normally take a cancellation token parameter.</span></span> <span data-ttu-id="c278f-116">每個取消權杖都有 <xref:System.Threading.CancellationTokenSource> ，它會建立權杖，並使用它來進行可取消的計算。</span><span class="sxs-lookup"><span data-stu-id="c278f-116">Each cancellation token has a <xref:System.Threading.CancellationTokenSource> that creates the token and uses it for cancelable computations.</span></span> <span data-ttu-id="c278f-117">常見的作法是使用長串的方法呼叫，將呼叫端的取消權杖傳遞給被呼叫端。</span><span class="sxs-lookup"><span data-stu-id="c278f-117">It is common practice to have a long chain of method calls that pass around the cancellation token from the callers to the callees.</span></span> <span data-ttu-id="c278f-118">因此，在可取消的計算中參與的大量方法最後會有解除標記參數。</span><span class="sxs-lookup"><span data-stu-id="c278f-118">Hence, a large number of methods that take part in a cancelable computation end up having a cancellation token parameter.</span></span> <span data-ttu-id="c278f-119">不過，取消權杖本身通常不會與這些方法中大部分的核心功能有關。</span><span class="sxs-lookup"><span data-stu-id="c278f-119">However, the cancellation token itself is not usually relevant to the core functionality of a majority of these methods.</span></span> <span data-ttu-id="c278f-120">將這類參數視為清單中的最後一個參數，會被視為良好的 API 設計作法。</span><span class="sxs-lookup"><span data-stu-id="c278f-120">It's considered a good API design practice to have such parameters be the last parameter in the list.</span></span>
